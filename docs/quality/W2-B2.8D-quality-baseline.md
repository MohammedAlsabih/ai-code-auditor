# W2-B2.8D Final — Post-Hardening Quality Baseline

A deterministic, offline, **no-AI** measurement of the auditor's findings on the
C1 (post-B2.8C1) corpus. Every number here comes from reading source and local
context by hand and classifying each finding across five independent axes. The
tool's output is **not** treated as ground truth; ambiguous cases are recorded
as `uncertain` rather than forced into a verdict.

This is a measurement-and-documentation round: **no rule, level, gate, scoring,
or precision changes were made.** Anything that looks like a defect is written
up as a recommendation for a later round, not fixed here.

The detailed per-finding labels (with paths, snippets, fingerprints, real

## Sample plan (deterministic — not probabilistic)

The sample is produced by the committed, path-agnostic sampler
`tools/quality_sample.py` (`select_sample(reports, cap=20)`), reproducible
from the two C1 `report.json` files by a fixed rule; it is **not** a random
draw and carries **no confidence interval**:

1. **Mandatory union.** Every finding with `level == error` or
   `gate_action == block` is reviewed in both repos (the 3 P002 blockers).
2. **Full small strata.** repo-b (57 findings) is reviewed in full. In either
   repo, any rule whose population is ≤ 20 is reviewed in full.
3. **Capped large strata.** For a rule with population > 20 in repo-a, 20
   findings are taken round-robin across projects in fingerprint order, after
   subtracting anything already taken by the mandatory union.
4. **Dedup on full identity.** `(alias, fingerprint, file, line)` — because a
   fingerprint deliberately excludes the line number, two findings that share a
   fingerprint but sit on different lines are distinct and both retained.

Selection is invariant to the JSON ordering of projects and findings
(everything is canonically sorted first), and real report paths are passed at
runtime only — they never appear in errors or output.

Result: **156 findings reviewed** (99 repo-a + 57 repo-b), of which 3 are
error/block.

## Ground truth binding (one label per finding)

Each of the 156 sampled findings carries exactly one local label
(schema_version 3), bound by a deterministic `sample_id` derived from the full
identity `(alias, project, file, line, rule, fingerprint)`. The label stores
the identity, all five axis values, and a non-empty free-text `evidence`
explaining the judgment — all of it local-only. `verify_one_to_one` enforces,
independent of ordering: no duplicate labels, no label outside the sample, no
sampled finding without a label, and literal identity/population agreement.
On both sides the stored `sample_id` is additionally **recomputed** from the
identity and must match — a forged id is rejected even if the same value is
planted in both files. The sample file is a **mandatory** input to
`build_public_summary`: there is no legal execution path that produces the
public summary without the one-to-one check.

## Five axes (kept separate by contract)

Detection validity, level appropriateness, gate appropriateness, and
actionability are **distinct questions** and are never collapsed into one score.

- **validity** — `confirmed | false_positive | uncertain`: is the thing the rule
  claims actually present in the source?
- **level_assessment** — `correct | too_high | too_low | uncertain`: is
  Error/Warning/Note appropriate?
- **gate_assessment** — `correct | too_strict | too_lenient | uncertain`: is
  Block/Review/Informational appropriate?
- **actionability** — `actionable | needs_context | non_actionable`: can a
  developer act on it as reported?
- **reason_code** — a short value from a fixed enum; aggregated publicly as
  per-rule `reason_code_counts` (the counts must sum to `reviewed`). The
  free-text evidence behind each code stays local.

Reason-code totals across the sample: complexity_over_threshold 52,

local_dev_credential 3, true_defect 2, public_by_design 1.

Reading these honestly:

- **Zero false positives in this sample** — but that is *not* a precision claim.
  The 15 `uncertain` findings are excluded from the confirmed-rate denominator
  precisely because the source alone could not settle them; had any resolved as
  false positives, the rate would drop. The confirmed rate reflects the decided
  subset only.
- **9.6% uncertainty** is concentrated entirely in the undeclared-import
  family (H002, H007) and three React-effect cases (R005) — see below.
- **actionable_rate is low (36.5%) by design, not by defect.** The bulk rules
  that dominate the corpus (P003 test credentials, P001 best-effort catches,
  R006 index keys) are correctly *detected* but are `needs_context` or
  `non_actionable` — they flag a real pattern that is often intentional. This is
  the axis separation working: a correct detection is not automatically an
  action item.

## Mandatory reviews

- **P002 (3 findings, all error/exact/block).** Confirmed legitimate. Each is a
  literal connection-string password committed to source: two localhost dev
  fallbacks in repo-a (one guarded by a production-environment throw, one in a
  design-time factory) and one localhost fallback in repo-b. A committed literal
  credential is a real hygiene issue regardless of host — it gets copy-pasted
  into production configs and leaks the DB pattern. `error/exact/block` is
  appropriate; reason `local_dev_credential`; actionable (move to env / user
  secrets). **No downgrade warranted.**
- **R005 (4 remaining after 14 → 4).** One is a genuine missing dependency
  (coordinates read inside an effect with `[]`, developer-acknowledged with an
  eslint-disable) — confirmed, actionable. The other three read a setter-like
  function *prop*; whether that prop is a stable state setter cannot be proven
  from the child file alone, so they are `uncertain` (`metadata_gap`), not
  false positives. Honest resolution: 1 confirmed, 3 uncertain.
- **H007 (2 remaining after ProjectReference closure).** Two OpenTelemetry
  sub-namespaces — the guessed package id equals the namespace and is absent
  from the registry, while the namespace is genuinely shipped by an
  OpenTelemetry meta/extension package. Cannot be settled without nupkg
  metadata → `uncertain` (`package_id_ne_namespace`). Suppression is now
  restricted to definite ProjectReference edges, which is why these two survive
  rather than being wrongly cleared.
- **H002 (9 findings).** All `uncertain` by design: five where the namespace is
  plausibly provided by a declared distribution under a different package id
  (`package_id_ne_namespace`) and four where a transitive provision graph is
  needed (`metadata_gap`). Per contract, id ≠ namespace is **not** decided
  without metadata.
- **N001 (1 finding).** A `NEXT_PUBLIC_*` maps browser key. The variable name
  alone does not prove a leaked secret — a maps browser key is public by design
  and controlled by domain restriction. The finding correctly stays a
  Warning/Review (`level`/`gate` correct) prompting a restriction check, but the
  secret verdict itself is `uncertain` (`public_by_design`) because the deployed
  key's restriction cannot be verified from source.

## Corpus and success criteria for measuring W3-B (AI single-finding review) later

When the AI review layer is measured, run it against the **same 156-finding
deterministic sample** recorded in `.quality-local/` — regenerable from the two
C1 `report.json` files with `tools/quality_sample.py`. The goal is **not** "to
resolve every uncertain finding"; honest uncertainty is a valid answer. The
pre-registered criteria:

1. **Agreement on decided cases.** On the 141 findings with a human
   confirmed/false-positive verdict, measure agreement with the human label.
2. **Confirmed-retention.** A currently-confirmed finding may only be flipped
   with new, valid evidence; unsupported flips count against the model.
3. **Calibrated abstention.** The model may answer `uncertain` /
   `needs_more_context`. Honest abstention on a genuinely undecidable case is
   **not** a failure and is scored separately from wrong verdicts.
4. **Evidence-gated resolution.** An `uncertain` finding (the 15: 9 H002,
   2 H007, 3 R005, 1 N001) counts as resolved only when the model produces new
   valid evidence — e.g. nupkg metadata for the provision graph, or cross-file
   context proving prop/setter stability — not merely a more confident guess.
5. **Citation validity.** Measure whether the evidence the model cites actually
   exists and supports the verdict, and whether the context it requested was
   necessary and sufficient.

No AI is run in this round.
